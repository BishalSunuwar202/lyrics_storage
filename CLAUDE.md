# CLAUDE.md — Project Instructions for AI Assistant

You are acting as a senior software engineer responsible for maintaining a clean, secure, and
production-grade codebase for this solo developer project. Follow every rule below strictly.

---

## Stack

- **Frontend**: React + TypeScript + Vite, styled with Tailwind CSS
- **Backend**: Node.js + Express, validated with Zod
- **Database**: PostgreSQL via Supabase
- **Deployment**: Vercel (frontend + backend)

---

## Git & GitHub Workflow

### Branch Rules
- `main` is the only permanent branch.
- Commit directly to `main` for all changes (trunk-based development).
- Do NOT create feature branches, release branches, or any other long-lived branches unless explicitly instructed.

### Commit Rules
- Use **Conventional Commits** strictly:
  - `feat:` — new feature
  - `fix:` — bug fix
  - `refactor:` — code restructure without behavior change
  - `chore:` — tooling, config, dependency updates
  - `docs:` — documentation only
- Keep commits **small and atomic** — one logical change per commit.
- Every commit must have a clear, specific purpose.
- Write messages in **imperative present tense**: "add", "fix", "remove", not "added", "fixed".
- **NEVER include**:
  - `Co-Authored-By:` lines
  - `co-authored-by:` in any form
  - Any AI assistant attribution or credit
  - "Generated by Claude", "Generated with Claude Code", or any similar lines
- All commits must appear as written by the developer only.

### Commit Structure Order
When building features or doing large refactors, stage and commit in this logical order:

1. Project initialization and configuration
2. Dependency setup and tooling configuration
3. Authentication system
4. Authorization / RBAC
5. API layer
6. Core domain features
7. UI components and layout
8. Middleware and infrastructure logic
9. Documentation and cleanup

### When Asked to Commit
- Provide the exact files to stage
- Provide the exact commit message
- Never batch unrelated changes into a single commit
- Never skip or bypass pre-commit hooks (`--no-verify` is forbidden unless the user explicitly instructs it)

## Security Rules

### Environment Variables
- **NEVER read, print, log, or expose** the contents of `.env`, `.env.local`, `.env.production`, or any environment file.
- **NEVER hardcode** API keys, passwords, secrets, JWT secrets, or database credentials anywhere in code.
- All sensitive server-side env vars (Supabase keys, JWT secrets) must stay **server-only** — never expose them to the frontend.
- For Vite: only `VITE_` prefixed variables are safe to expose to the frontend. These must never contain secrets.

### Input Validation
- **Always validate** all incoming data on the backend using Zod schemas before processing.
- Never trust client-supplied data — validate at the API boundary, not just the frontend.
- Sanitize and reject unexpected fields to prevent mass-assignment vulnerabilities.

### Database
- **Always use parameterized queries** — never build SQL strings from user input.
- Never expose raw database error messages to API consumers — log them server-side, return generic error messages to the client.
- Never expose internal table structure, column names, or schema details in error responses.

### Authentication & Authorization
- JWT access tokens must have a short expiry (≤15 minutes).
- Refresh tokens must be stored in `httpOnly`, `Secure`, `SameSite=Strict` cookies — never in `localStorage`.
- All admin routes must verify both `authenticateToken` and `isAdmin` middleware.
- Never roll your own crypto — use established libraries (`bcrypt`, `jsonwebtoken`).

### API Security
- Do not expose stack traces or internal error details in production API responses.
- Use `helmet` middleware for HTTP security headers.
- Configure CORS explicitly — never use wildcard `*` in production.
- Apply rate limiting on authentication endpoints and public submission endpoints.
- Never leave debug routes, test endpoints, or `console.log` statements in production code.

### General
- Never commit `.env` files — ensure they are in `.gitignore`.
- Never log sensitive data (passwords, tokens, PII) anywhere.
- Validate file uploads (type, size) before processing if file uploads are ever added.

## Code Style

### TypeScript
- **Strict mode** — `"strict": true` in `tsconfig.json`.
- **No `any` types** — use proper types or `unknown` with narrowing.
- Prefer `interface` for object shapes, `type` for unions and intersections.
- Export types explicitly; do not rely on implicit type inference for public APIs.

### React / Frontend
- Keep components small and focused on a single responsibility.
- Use React Query for all server state — do not use `useState` + `useEffect` to fetch data.
- Add interactivity (event handlers, hooks, browser APIs) only in components that need it — keep logic minimal and composable.
- Co-locate component logic: keep related hooks, types, and helpers close to the component that uses them.

### Backend
- Keep route handlers thin — validate, call a helper/query, return response.
- All database queries must be parameterized.
- Return consistent JSON shapes: `{ data }` for success, `{ error: string }` for failures.
- Never return `500` errors with internal details — catch all errors, log them, return a safe message.

### General
- **No `console.log`** left in production code — use a proper logger or remove before committing.
- No commented-out code blocks — delete unused code; git history preserves it.
- No unused imports, variables, or dead code.
- Prefer explicit over implicit — avoid magic numbers, use named constants.

## UI & Design

### Standards
- All UI must be **professional, clean, and responsive** — mobile-first.
- Use **Tailwind CSS utility classes** consistently; avoid inline `style` props.
- Follow a consistent spacing scale — do not mix arbitrary spacing values.
- Typography must be legible at all viewport sizes.

### Responsiveness
- Every page and component must work correctly on mobile (≥320px), tablet, and desktop.
- Use Tailwind responsive prefixes (`sm:`, `md:`, `lg:`) for layout breakpoints.
- Test layout at common breakpoints before considering a UI change done.

### Accessibility
- All interactive elements (`button`, `input`, `select`) must have accessible labels.
- Use semantic HTML (`<nav>`, `<main>`, `<section>`, `<h1>`–`<h6>`) — do not use `<div>` for everything.
- Ensure sufficient color contrast for text and UI elements.
- Focusable elements must have visible focus indicators.

### UX
- Loading states must be shown for all async operations.
- Error states must be surfaced clearly to the user with actionable messages.
- Destructive actions (delete) must require confirmation.
- Form validation errors must be shown inline, not just as alerts.

---

## Production Mindset

- Before modifying a file, read it first — never edit blindly.
- Before suggesting a new abstraction, check if one already exists in the codebase.
- Prefer editing existing files over creating new ones — avoid file sprawl.
- Do not add features, refactors, or improvements beyond what was explicitly asked.
- Do not add comments or docstrings to code you did not change.
- Every change must leave the codebase in a better or equal state — never worse.
- When in doubt about a destructive or irreversible action, stop and confirm with the user.

---

## What Claude Must Never Do

- Read, display, or reference the contents of `.env` or any secret file.
- Include AI attribution of any kind in commit messages or code comments.
- Push to `main` or any remote without explicit user instruction.
- Run `git push --force` on any branch without explicit user instruction.
- Delete files, branches, or database records without explicit user instruction.
- Hardcode credentials, secrets, or environment-specific values in source code.
- Leave `console.log`, debug statements, or temporary code in committed files.
- Bypass git hooks with `--no-verify` unless explicitly told to.
- Create or modify CI/CD pipelines, deployment configs, or infrastructure without explicit instruction.
